## Tier 0 -- Ready now: the fly-through stutter

The biggest measured hitch, already diagnosed this session. Do the cluster in order; **step EV1
alone removes the spike**. The other steps reduce the steady cost and the VRAM it pins.

### [ ] EV. Deferred-eviction cost (the fly-through stutter)

Multiple agents converged here -- it is the #1 measured stutter. Sub-approaches span a same-day budget cap up to a GPU-resident readiness hierarchy.

**EV1. Deferred-eviction step rescans the ENTIRE pending_evict set every frame (the fly-through stutter)**  
`impact: high` * `effort: medium` * `source: arch-editor`

- *Now:* schedule_bakes step 4 (bake_scheduler.rs:683-700) iterates the WHOLE pending_evict map every frame, calling replacement_ready() -- which runs up to (lod_count-1) BVH any_overlap queries plus per-LOD brick-key hash lookups (replacement_ready/coarser_region_has_geometry, bake_scheduler.rs:297-344) -- for EVERY deferred chunk. Nothing bounds the per-frame work; every entry is re-tested from scratch even though almost none flip to ready on a given frame.
- *Evidence:* bake_scheduler.rs:683-700 (loop over sched.pending_evict, replacement_ready per entry); bake_scheduler.rs:297-344 (replacement_ready does up to lod_count BVH/brick queries); EVICT_DEFER_CAP=120 at bake_scheduler.rs:184; trace attribution ~14us/entry x 9.6k.
- *Approach:* Two complementary changes. (1) Budget + round-robin: keep a rotating cursor into pending_evict and test only K entries/frame (e.g. 256), so per-frame cost is O(K) not O(N); a held chunk that isn't ready this frame simply gets re-checked a few frames later (its age counter still increments toward EVICT_DEFER_CAP so nothing pins forever). Maintain pending_evict as a per-coarse-cell index (HashMap<coarse BrickKey, Vec<finer ChunkKey>>) so a newly-resident coarse brick directly yields its now-satisfied dependents in O(1). This turns the whole step from O(N) sweep into O(newly-baked) work. Combine with the round-robin as a backstop for the 'no coarser geometry will ever come' (coarser_region_has_geometry==false) case.
- *Risk:* Make-before-break correctness: must guarantee a chunk is never dropped before its replacement is resident, and never pinned forever. The age-cap backstop and the existing replacement_ready predicate preserve the invariant; the index just changes WHEN the predicate is evaluated. Off-by-one in the coarse-cell mapping could drop a chunk one frame early (transient hole) -- cover with the existing replacement_ready_gates_on_coarser_residency test plus a new index-consistency test.
- *Verified:* confirmed -- Every cited claim checks out against the code. bake_scheduler.rs:681-686 loops the WHOLE pending_evict HashMap each frame (no cursor, no per-frame cap) calling replacement_ready for every entry. replacement_ready (290-300) is genuinely expensive in the common deferred case: coarser_region_resident (306-316) does up to (lod_count-1) hash lookups, and on a miss coarser_region_has_geometry (322-337) 
- *Refs:* Make-before-break LOD handoff (geometry clipmaps, Hoppe 2004); event-driven dependency tracking / dirty-wake instead of per-frame polling.

### [ ] C4. Whole-set re-dirty on add/remove re-enqueues every resident-window chunk at all 8 LODs
`impact: medium` * `effort: medium` * `area: algorithm` * `source: cpu-scheduler`

- *Now:* When set_changed (any add/remove) or rebake_all, schedule_bakes re-dirties EVERY chunk in every LOD's window (bake_scheduler.rs:583-591): 8 LODs x R^3 chunks (R = ring_bricks/CHUNK_BRICKS = 32 at the new 2x ring -> up to 8x32^3 ~ 262k chunk inserts into the pending HashSet). A single spawned or deleted primitive thus dirties the entire resident clipmap, then the bake's content-hash cache has to re-classify all of them to discover almost nothing changed.
- *Evidence:* bake_scheduler.rs:540-542 set_changed triggers on count mismatch or any new key; :583-591 the whole-ring re-dirty loop over all LODs x chunk_window_keys; ring_chunks_per_axis :264-266
- *Why:* Spawning/deleting one object (common in the editor) floods pending with the entire clipmap, spilling over many frames and re-running classify on ~all resident bricks even though the hash skip will keep most. The hash cache saves the bake but not the per-brick cull+hash cost.

A removed edit's AABB comes from prev_aabbs.map; an added edit's from current. Only fall back to whole-ring on an actual rebake_all (material registry change / explicit invalidation). This makes spawn/delete cost O(footprint), matching the move path.
- *Risk:* Must capture the removed entity's last AABB before it's dropped from prev_aabbs (it currently is, via the map). Edge case: an edit whose order change reshuffles material resolution of neighbors -- but CSG order only affects overlapping edits, whose footprints overlap anyway, so the footprint dirty covers it.
- *Refs:* Incremental dirty-region invalidation

### [ ] C5. Chunk row insert/remove re-stamps and re-uploads the entire tail of the sorted row array (O(rows) per topology event)
`impact: medium` * `effort: large` * `area: data-structure` * `source: data-structures`

- *Now:* `LiveChunkTables::set_brick`'s new-chunk path inserts into a Vec at the sorted position and then re-stamps key_to_row + marks dirty for EVERY row at/after the insert (chunk.rs:332-337); `clear_brick`'s empty path does the symmetric tail re-stamp (chunk.rs:368-373). Because `tile_run_base = slot * TILE_RUN_SLOT` is slot-stable, the ROW's `tile_run_base` does NOT actually change when rows shift -- only `key_hi/key_lo/occ` matter, and those are identical for a shifted row. Yet every shifted row is marked dirty and re-uploaded. A new chunk near the front of the key order dirties (and re-uploads) thousands of unchanged rows.
- *Evidence:* chunk.rs:332-337 (insert re-stamps + dirties whole tail); chunk.rs:368-373 (remove re-stamps whole tail); lookup_at chunk.rs:417-423 shows a shifted row's contents are key+occ+slot-base -- unchanged by the shift itself; delta upload render.rs:1278-1282 re-uploads each dirty row
- *Why:* During a coarse LOD snap, many chunks enter/exit and each entry/exit can dirty O(rows) entries -- turning the 'incremental' delta into near-full re-uploads on exactly the frames that matter (the snap stutter). The shifted rows' GPU bytes are identical to what's already there.
- *Approach:* The sorted-array row order is the real cost driver. Replace the binary-searched sorted Vec with a layout that does NOT shift on insert/remove: e.g. a GPU hash table keyed by `chunk_gpu_key` (open-addressing, the shader probes instead of binary-searches), so a new chunk writes ONE slot and shifts nothing. The chunk count is bounded (resident window), so a fixed-capacity hash with linear probing is cheap and removes ALL tail re-stamping. Alternatively keep the sorted array but only mark a shifted row dirty when its CONTENT (key/occ/base) actually differs from what the GPU holds -- track a per-row last-uploaded signature and skip identical re-uploads (cheap, low-risk interim).
- *Risk:* A GPU hash table changes the shader's `find_brick_lookup` from binary search to probing -- needs the bindings.wgsl mirror + the differential churn test (chunk.rs:927) updated, and load-factor/tombstone handling. The interim 'skip identical row' optimization is low-risk but only cuts upload bytes, not the CPU re-stamp loop.
- *Refs:* GPU open-addressing hash (e.g. NanoVDB/Tree64 root hash table); two-level grid

### [ ] C6. sort_drained sorts the full drained set every frame with per-key distance + chunk_world_size recomputation
`impact: low` * `effort: small` * `area: cpu` * `source: cpu-scheduler`

### [ ] G1. Cone prepass seeds only a per-tile START distance; it does not export a per-tile LOD floor or coarse hit, so each pixel still re-walks all LODs from the seed

### [ ] G2. Shrink the 3-target Rgba16Float G-buffer (bandwidth + the emissive target is wasteful)
`impact: medium` * `effort: medium` * `area: gpu` * `source: gpu-pipeline`

- *Now:* The primary pass writes THREE full-res Rgba16Float MRT targets (render.rs:273 GBUFFER_FORMAT, gbuffer_targets:1093-1102): albedo+dist, octNormal+metallic+roughness, and emissive+sun_vis. The third target (emissive) is zero for the overwhelming majority of pixels -- only the handful of emissive demo objects ever write nonzero rgb -- yet a full Rgba16Float attachment is cleared, written, and sampled every frame for the whole screen. albedo.a stores camera distance as fp16, which loses precision for world-pos reconstruction at range.
- *Evidence:* render.rs:584-609 binds all three as RenderPassColorAttachment with Clear+Store; sdf_raymarch.wgsl:138-142 always writes emissive (mostly vec4(0,0,0,sun_vis)); sdf_deferred_lit.wgsl:58-60 samples emissive every pixel.

- *Approach:* Pack to 2 targets: target0 = albedo(rgb)+packed metallic/roughness; target1 = octNormal(rg)+sun_vis(b)+a small emissive-present flag, and fetch emissive radiance by material id in the lit pass (it already imports material_at -- the lit pass currently doesn't, but it samples a tiny material table). Switch color attachment load to a sky-sentinel-aware path so clears can be dropped where coverage is guaranteed.
- *Risk:* Reconstructing emissive from material id in the lit pass requires binding the material table there (minor layout change). Octahedral normal at fp16 is fine; packing metallic/roughness into 8-8 of an fp16 needs care.
- *Refs:* Deferred G-buffer packing (Crytek/DOOM-style); octahedral normal encoding (already used).

### [ ] G3. Eliminate per-frame bind-group, material-buffer, and dummy-buffer (re)creation in render nodes
`impact: medium` * `effort: medium` * `area: gpu` * `source: gpu-pipeline`

- *Now:* Every render-graph node rebuilds all of its bind groups from scratch every frame inside run(): GBuffer node creates bg0/bg1/bg2 (render.rs:521-577), Combine creates bg0/bg1 (684-693), Cone creates bg0/bg1/bg2 (2168-2212). bg1 (the atlas) has 12 bindings and is rebuilt 2x per frame (cone + gbuffer) even on frames where no atlas resource changed. - *Evidence:* render.rs:540-569 (12-entry bg1 built in gbuffer run), 2176-2205 (same 12-entry bg1 rebuilt in cone run), 1310-1350 (material buffer recreated unconditionally each frame), 521-535 (camera bg0 rebuilt per frame).

- *Approach:* Resize the cone seed texture in a prepare system keyed off the same view size as prepare_sdf_gbuffer (round up to tiles), reusing the existing resize-on-change pattern. Keep a max cap to avoid churn on drag-resize.

### [ ] M6. Bake staging output buffers grow-only and sized to peak job count
`impact: low` * `effort: small` * `area: memory` * `source: memory-vram`

- *Now:* The bake dist/mat output staging buffers are sized to n jobs x per-tile u32s and only ever recreated when too small (render.rs:2346-2365, needs_dist/needs_mat = size < required). A single large-edit frame that dirties 70k+ bricks (the comment at render.rs:2462 cites 70k) sizes dist_buffer to 70kx256u32x4 = 70MB and mat_buffer to 70kx512u32x4 = 140MB -- ~210MB of staging VRAM that is then held for the rest of the session even when subsequent frames bake a handful of bricks. There is also a GPU_BAKE_JOB_CAP ceiling but the buffers persist at whatever peak they reached.
- *Evidence:* render.rs:2346-2365 (size = nxTILE_U32x4, grow-only via is_none_or size<), render.rs:358-368 (BAKE_DIST_TILE_U32=512, BAKE_MAT_TILE_U32=1024), render.rs:2462-2463 ('a big sphere dirties 70k+ bricks')
- *Why:* Bounded by GPU_BAKE_JOB_CAP and the per-frame budget, but a one-time large edit permanently reserves up to ~200MB of staging that is idle thereafter.
- *Approach:* Cap the staging buffers at the soft per-frame bake budget (SOFT_BAKE_BUDGET / GPU_BAKE_JOB_CAP) rather than peak observed n, and shrink them back to a baseline when several frames pass with a much smaller job count (recreate-smaller, same pattern as the grow check). Since jobs are already capped per frame to the budget, the buffer never legitimately needs to exceed budgetxtile size.
- *Risk:* Low -- recreating a smaller buffer is the same code path as growing; just add a shrink condition with hysteresis so it doesn't thrash.

### [ ] M7. Bake dist output buffer wastes 50% to row padding (256-byte copy alignment)
`impact: low` * `effort: medium` * `area: memory` * `source: memory-vram`

- *Now:* The distance staging tile is BAKE_DIST_ROW_U32=64 u32/row but only 32 are real R16 data (32 real + 32 pad/row per the comment), padded so copy_buffer_to_texture sees bytes_per_row=256 (render.rs:358-364, :2488). So half of every dist tile's staging bytes -- and half the buffer->texture copy bandwidth -- is zero padding. Over a 70k-brick bake that is ~35MB/frame of pure padding copied.
- *Evidence:* render.rs:358-364 (BAKE_DIST_ROW_U32=64, '32 real + 32 pad per row'), render.rs:2486-2490 (bytes_per_row 256)
- *Why:* Halves dist staging efficiency, but dist is the smaller of the two buffers (mat is 4x larger and already unpadded); affects bandwidth on big bake frames only.
- *Approach:* A dist tile row is 64 R16 texels = 128 bytes, which violates the 256-byte bytes_per_row requirement. Pack two brick rows per 256-byte copy row, or batch tiles so the copy region is wider (copy a strip of N tiles at once with a 256-aligned combined row) instead of one tile at a time, eliminating the per-tile pad. Alternatively keep dist in the mat buffer's already-512-byte rows by interleaving. 
- *Risk:* Changing the buffer->texture layout must keep the bake shader's write offsets and tile_origin packing in sync (render.rs:1165, chunk.rs:133); covered by the shader-validation test rig but needs careful offset math.
- *Refs:* WebGPU/wgpu copyBufferToTexture 256-byte bytesPerRow constraint.

## Tier 5 -- Research & experimental (big bets)

### [ ] R1. Segment tracing (local Lipschitz bounds) to take longer safe steps than sphere tracing
`impact: medium` * `effort: medium` * `area: shader` * `source: research-sdf`

- *Now:* The in-brick march is Keinert over-relaxation sphere tracing (march.wgsl:186-223): step = omega·d with an unbounding-sphere overshoot validation and undo. Over-relaxation helps but still validates per-step and falls back to plain d on the blended (non-eikonal) field where it forces omega=1 (march.wgsl:218), losing acceleration exactly in the LOD cross-fade band.
- *Evidence:* march.wgsl:186-223 (Keinert over-relax + fallback), march.wgsl:218 local_omega forced to 1 while blending. Research: Segment Tracing Using Local Lipschitz Bounds (Galin et al., Eurographics 2020, CGF 39(2):545-554) -- compute a local Lipschitz bound over a segment to enlarge the safe step with no acceleration structure and no fallback.
- *Why:* On the trilinear brick field a per-brick (or per-segment) Lipschitz constant is cheap to bound (gradient magnitude of trilinear interpolation is analytic), letting steps exceed plain d safely even in the cross-fade band where over-relaxation is currently disabled. Fewer iterations than over-relaxation on curved/grazing regions, and it removes the validate-and-undo branch divergence.
- *Approach:* Precompute or bound a per-brick Lipschitz constant L (max trilinear gradient magnitude, derivable from the 8 corner values) and store it alongside the brick. In march.wgsl, replace omega·d with d/L_local where L_local is the segment Lipschitz bound; keep it active during blending (it is correct on non-eikonal fields, which is the whole point of segment tracing). Reuse the existing brick-exit cap.
- *Risk:* Bound must be a true local upper Lipschitz bound or steps overshoot; near brick seams the bound must cover the apron. Lower risk than it sounds because the brick field is piecewise-trilinear (closed-form gradient). Replaces a working over-relaxation path, so A/B against step-count debug view.
- *Refs:* Galin et al. 2020, onlinelibrary.wiley.com/doi/abs/10.1111/cgf.13951; code github.com/aparis69/Segment-Tracing

### [ ] R2. Frame-coherent reprojected hierarchical Z / coarse-to-fine cone seeding (extend the existing 8x8 cone prepass)
`impact: medium` * `effort: medium` * `area: gpu` * `source: research-sdf`

- *Now:* There is a single-level cone prepass (sdf_cone_prepass.wgsl): one 8x8 tile cone march writes a per-tile seed_t that the per-pixel pass starts from. It is one level deep, recomputed from t=0 every frame, and discards last frame's hit distances entirely.
- *Evidence:* sdf_cone_prepass.wgsl:1-125 (single TILE=8 level, marches from t=0), march.wgsl seeds via start_t. Research: 'cone marching' multi-pass (pointersgonewild.com 2026 recursive SDF, reindernijhoff.net WebGPU SDF editor 2026); Skipping Spheres (EG UK 2024) low-res buffer seeding; standard Hi-Z reprojection.
- *Why:* A single 8x8 level leaves a large gap between tile-cone resolution and per-pixel; a 2-3 level cone pyramid (e.g. 32x32 -> 8x8 -> pixel) skips far more empty corridor per pixel. Reprojecting last frame's hit distance as an additional seed lower bound (camera moves are small frame-to-frame in a fly-through) cuts steps further at near-zero cost and directly attacks primary-ray iteration count.
- *Approach:* Make the cone prepass a 2-3 level mip cascade (coarse tile feeds finer tile feeds pixel), each starting from the coarser seed. Additionally store last frame's per-pixel hit t in a history texture, reproject by the camera delta, and use max(cone_seed, reprojected_t - guard) as start_t. Both are pure lower bounds, so the per-pixel march result is unchanged.
- *Risk:* Reprojection guard band must be conservative (disocclusion -> reprojected seed invalid -> fall back to cone seed); needs a validity test. Multi-level cone adds a compute pass. Low correctness risk (seeds are lower bounds only).
- *Refs:* Skipping Spheres EG UK 2024, diglib.eg.org/server/api/core/bitstreams/0dd5043c-...; cone marching pointersgonewild.com/2026-03-06-a-recursive-algorithm-to-render-signed-distance-fields/

### [ ] R3. SDF-traced glossy reflections (SSR + SDF fallback) reusing the march and existing SSR plumbing
`impact: medium` * `effort: medium` * `area: shader` * `source: research-gi-bevy`

- *Now:* render.rs:62-64 documents an SSR reflection path (prev_clip_from_world reprojects a reflected world point into last frame's screen), but the lit pass (sdf_deferred_lit.wgsl:103) outputs only diffuse sun+emissive -- no specular indirect. Metallic/roughness are in the G-buffer (normal_mat.b/.a) but never used for reflections.
- *Evidence:* src/sdf_render/render.rs:62-64 (SSR reprojection matrix already plumbed); assets/shaders/sdf_deferred_lit.wgsl:54-57,103 (metallic/roughness read but unused for specular); assets/shaders/sdf/march.wgsl:44-50 (cone_k for roughness-widened cones)
- *Why:* Big perceived-quality jump (wet/metal/water surfaces) for modest cost since rough reflections can sample the DDGI probes directly (no extra rays) and only sharp reflections trace. Reuses the march + SSR reprojection already designed.
- *Approach:* Add a reflection pass: for each pixel, reflect the view ray about the G-buffer normal, screen-trace first (SSR using prev frame), and on SSR miss fall back to one `raymarch` along the reflection ray, shading the hit with the same sun+emissive(+probe) as DDGI. Roughness drives cone width via the already-present MarchQuality.cone_k. This is exactly Lumen's 'reuse the radiance cache for rough, dedicated rays for sharp' split, and VCT/SDFGI both note specular is a free add-on once you have an SDF tracer.
- *Risk:* Sharp reflections are noisy without temporal accumulation; needs a history buffer + neighborhood clamp. SSR-to-SDF handoff must be seamless to avoid a visible ring at screen edges.
- *Refs:* Lumen reflections (https://dev.epicgames.com/documentation/en-us/unreal-engine/lumen-technical-details-in-unreal-engine); Crassin VCT specular (https://arxiv.org/pdf/2104.00618).

### [ ] R4. Move the GI/probe trace and bake-classify onto the render-graph async-compute timeline, not the Update schedule
`impact: medium` * `effort: medium` * `area: architecture` * `source: research-gi-bevy`

- *Now:* The frame is MAIN-THREAD bound inside Update (per the trace): schedule_bakes, gather_sorted_edits, dispatch_bake all run there. The bake already offloads classify to AsyncComputeTaskPool (bake_scheduler.rs:1271,1666) but the GPU-side fill is a render-graph compute node (SdfBrickBakeLabel) sequenced strictly opaque->bake->cone->gbuffer->lit (render.rs:838-848). Any future GI trace, if added naively to Update or as a serial graph node, would extend the same critical path.
- *Evidence:* src/sdf_render/render.rs:824-848 (serial graph edges); src/sdf_render/bake_scheduler.rs:1271 (async classify already in use as the pattern to follow); Bevy PipelineCache async-by-default + get_render_pipeline Option accessor
- *Why:* Keeps a heavy new GI pass off the already-saturated main thread / serial graph, exploiting Bevy's pipelined render world. One-frame GI latency is imperceptible for diffuse indirect.
- *Approach:* Bevy compiles pipelines async on AsyncComputeTaskPool by default and the render graph runs after extract/prepare/queue. Place the DDGI/probe trace as a render-graph compute node that is NOT on the gbuffer->lit dependency chain for the CURRENT frame -- let it consume the PREVIOUS frame's G-buffer/atlas and feed THIS frame's lit pass (one-frame-latent GI, standard for DDGI). That removes it from the serial per-frame critical path entirely. Use get_render_pipeline() (Option) not unwrap() in the node so a not-yet-compiled GI pipeline can't panic during async warmup.

- *Refs:* Bevy PipelineCache docs (https://docs.rs/bevy/latest/bevy/render/render_resource/struct.PipelineCache.html); Bevy render architecture (https://bevy-cheatbook.github.io/gpu/intro.html).

### [ ] R5. Tree64 / nested-DDA two-level brickmap traversal as an alternative to per-step LOD-walk + binary-searched chunk table
`impact: medium` * `effort: large` * `area: algorithm` * `source: research-sdf`

- *Now:* Chunk lookup is a sorted-key binary search (find_chunk, brick.wgsl:96-119) over the whole chunk_buf each probe, memoised per-ray by a per-LOD ChunkCache (brick.wgsl:397-426). resolve_march walks all LODs fine->coarse every step (brick.wgsl:488-517). So each step is up to lod_count binary searches (cached to ~O(1) within a chunk). The structure is a flat sorted array + popcount tile-run, not a hierarchical grid.
- *Evidence:* brick.wgsl:96-119 (binary search), :397-517 (per-LOD cache + fine->coarse walk). Research: dubiousconst282 'A guide to fast voxel ray tracing using sparse 64-trees' (Oct 2024) -- 4^3 contree / Tree64; VoxelRT MultiDDA two nested DDA loops (8^3 then voxel) beats SVO by up to 60%; expenses/tree64 Rust port exists.
- *Why:* A binary search over the full chunk array is O(log N) per uncached probe and scales with total resident chunks; a hierarchical 4^3 occupancy grid (or NanoVDB-style root/internal/leaf bitmask) gives O(1) descent and natural empty-space skipping that subsumes the manual chunk-DDA. On incoherent rays (shadows, the removed-then-future GI) the binary search + per-LOD walk is where divergence hits. This is a structural simplification, not just a speedup.
- *Approach:* Replace the sorted chunk array + per-LOD walk with a single sparse occupancy hierarchy keyed in world space: a coarse 3D index grid (or Tree64 nodes) whose cells point at chunk rows, traversed by one HDDA/nested-DDA loop that descends only where occupied. The brick tile-run + R16Snorm atlas stay as the leaf payload (this is exactly the brickmap leaf). A Rust reference (expenses/tree64) maps onto the existing chunk.rs delta-upload model. Keep the clipmap LODs as the coarse levels.
- *Risk:* Large rewrite of brick.wgsl traversal and chunk.rs upload; must preserve the absolute-addressing and apron-trilinear-seam correctness the current code carefully guarantees (brick.wgsl:42-59 comments). High effort/risk; best treated as an R&D spike validated against the existing GPU rig (tests/sdf_gpu_rig.rs).
- *Refs:* dubiousconst282.github.io/2024/10/03/voxel-ray-tracing/; github.com/dubiousconst282/VoxelRT; github.com/expenses/tree64; NanoVDB ACM 10.1145/3450623.3464653

### [ ] R6. DDGI / SDFGI irradiance-probe volume reusing the existing SDF march (best GI fit)
`impact: medium` * `effort: large` * `area: gpu` * `source: research-gi-bevy`

- *Now:* The deferred lit pass (assets/shaders/sdf_deferred_lit.wgsl:8-10,103) computes ONLY analytic sun + emissive: `let lit = direct + emissive`. There is no indirect bounce at all -- the radiance-cascade GI path was removed. The shader comment and render.rs:836-837 already point at `plans/sdf-ddgi-probe-volume.md` as the intended replacement, but it is not implemented.
- *Evidence:* assets/shaders/sdf_deferred_lit.wgsl:8-10,94-103; assets/shaders/sdf/shadows.wgsl:29-60; assets/shaders/sdf/oct.wgsl; assets/shaders/sdf/march.wgsl:45-50,67; src/sdf_render/render.rs:836-837,858-861
- *Why:* This is the single biggest visual-quality win available and the one the architecture is already shaped for. DDGI is O(1) at shade time (one probe fetch in the lit pass), amortized over frames, and needs NO RT hardware. SDF-DDGI papers report <5ms on a GTX 970M / <1ms on a 2080Ti for the trace+update. Because tracing reuses the existing march and the cascades already exist, incremental cost is a compute pass with a fixed ray budget, fully decoupled from screen resolution.
- *Approach:* Implement Dynamic Diffuse Global Illumination (Majercik/McGuire JCGT 2019), specialized to this SDF scene exactly as Godot's SDFGI and Flax's software-RT DDGI do. The codebase ALREADY has every primitive: (1) a robust secondary SDF tracer -- `soft_shadow`/`surface_shadow` in assets/shaders/sdf/shadows.wgsl:29-60 already marches arbitrary secondary rays through the field with a per-ray chunk cache; (2) octahedral encode/decode in assets/shaders/sdf/oct.wgsl (oct_encode/oct_decode) for probe irradiance + depth atlases; (3) a natural clipmap cascade structure (8 LODs, camera-centred rings) that maps 1:1 onto DDGI/SDFGI cascades that scroll with the camera; (4) the G-buffer already reconstructs world position from albedo.a=distance (sdf_deferred_lit.wgsl:45-46,94-97) and stores oct normal + material, so the receiver side is free. Plan: a compute pass per cascade of a 3D probe grid (e.g. 32^3 probes/cascade, 8x8 oct texels each) traces N rays/probe/frame via `raymarch` (reusing MarchQuality with a coarse lod_floor + capped steps -- that struct in sdf/march.wgsl:45-50 was literally built for secondary rays), shades each hit with sun*sun_vis + emissive (the material table is already a render-world storage buffer, render.rs:858-861), blends into an octahedral irradiance atlas + a depth atlas for Chebyshev visibility (kills leaks; SDF thickness guarantees walls occlude). ### [ ] R9. Replace per-frame queue.write_buffer copies with wgpu write_buffer_with staging-slice writes
`impact: low` * `effort: small` * `area: memory` * `source: research-gi-bevy`

- *Now:* The incremental chunk-table upload encodes Rust Vecs then calls queue.write_buffer for each changed run (render.rs:1632,1664,1676). Each write_buffer makes an extra copy from the encoded Vec<u8> into wgpu's staging buffer. The session note already shows Extract is cheap (0.85ms), but per the Bevy rendering-optimization work, write_buffer's double-copy is a known avoidable cost as table churn grows on LOD snaps.
- *Evidence:* src/sdf_render/render.rs:1611-1676 (write_buffer delta path); Bevy renderer-optimization tracking issue #12590 documents the write_buffer double-copy remedy
- *Why:* Saves one memcpy per uploaded region; meaningful only on heavy-churn snap frames where many regions upload. Modest but cheap and low-risk; complements the bigger eviction/change-detection fixes.
- *Approach:* Switch the hot delta uploads (chunk lookup rows, tile-run regions) to wgpu's write_buffer_with() / Bevy's RawBufferVec-style staging-slice API: request a mutable slice into the staging buffer and serialize GpuChunkLookup/GpuBrickTile directly into it, skipping the intermediate Vec<u8>. This is the exact remedy Bevy applied in its renderer-optimization tracking work for per-instance data.
- *Risk:* write_buffer_with requires the exact byte length up front and correct alignment; mis-sized slices corrupt the buffer. Encase/ShaderType layout must match the manual write exactly.
- *Refs:* Bevy renderer optimization tracking (https://github.com/bevyengine/bevy/issues/12590); wgpu Queue::write_buffer_with.

### [ ] R10. Dreams-style splat / compute-rasterized surfels as an alternative primary-visibility path for distant LODs
`impact: low` * `effort: large` * `area: architecture` * `source: research-sdf`

- *Now:* All primary visibility is full sphere-trace raymarch into the deferred G-buffer (sdf_raymarch.wgsl + march.wgsl) at every pixel and every LOD, including distant coarse LODs where a pixel covers many voxels and the march still iterates.
- *Evidence:* march.wgsl raymarch is the sole primary path; eff_lod/blend logic (march.wgsl:143-184) shows distant pixels already collapse to coarse LODs. Research: Media Molecule Dreams (Evans, SIGGRAPH 2015 'Learning from Failure') renders dense point splats extracted from the SDF via compute-shader rasterization rather than tracing primary rays; 'a pure SDF raytracer has zero overdraw so g-buffering brings little, but a point renderer can gain' (Beyond3D/HN threads).
- *Why:* For the coarsest clipmap shells (far terrain), one compute-rasterized splat per resident surface voxel can be cheaper and more temporally stable than per-pixel tracing, and it sidesteps grazing-angle step blowup. But it is a major architectural fork from the current clean trace+deferred pipeline and overlaps with the cheaper max-mipmap win; include as a longer-horizon option.
- *Approach:* For the 2-3 coarsest LODs, extract a sparse point set (surface voxels with normal/material) at bake time and compute-rasterize them into the same G-buffer (atomic depth, Dreams/AMD-style tiled splatting), reserving the sphere-trace for near LODs. The deferred lit pass is unchanged. Hybrid: trace near, splat far, blend in the existing LOD cross-fade band.
- *Risk:* Splat artifacts at silhouettes (Dreams shows these), G-buffer write contention, and a second visibility codepath to keep consistent with the trace. High effort; only justified if max-mipmap + segment tracing don't close the far-LOD cost. Speculative.
- *Refs:* Alex Evans 'Learning from Failure' SIGGRAPH 2015; Beyond3D Dreams SDF thread; Rendering Point Clouds with Compute Shaders (arxiv 1908.02681)
